XML structure¶

To use the mBlox XML interface, all of your requests to, and responses from, the mBlox platform must be structured in four types of XML documents:

  1. You send the Notification Request to the mBlox platform with the MT message for the subscriber.
  2. The mBlox platform returns a Notification Request Result to you with a status on your MT message request.
  3. The mBlox platform sends your request on to the operator’s network, which returns a Delivery Receipt with a status on the MT message delivery. The Delivery Receipt is converted to XML and sent back to you in the Notification Service.
  4. When a subscriber sends an SMS message to your account via the operator network, the mBlox platform converts the MO message to XML and sends it to you in the Response Service.

You can see how these documents are used in the XML interface workflow by reading the XML interface overview.

The following sections contain both the details of each XML document and its DTD. Click on any line of XML in the code inspectors to see details about its implementation.

    Version CDATA "3.5"

>

Note that there are three related result codes in the Notification Request Result:

  • <RequestResultCode> - contains a result related to the XML format or header parameters of the original Notification Request
  • <NotificationResultCode> - contains a result related to whether the parameters in the <Notification> element of the original Notification Request have the correct syntax
  • <SubscriberResultCode> - contains a result related to the whether the individual MT message has been accepted by the mBlox platform

The lower-level code only occurs if the higher-level code does not indicated an error. For example, the <SubscriberResultCode> element is only present in the XML document if the <NotificationResultCode> corresponds to “OK”.

A retry parameter indicates whether your application should attempt to retry a Notification Request which has returned an error.

<RequestResultCode> and <NotificationResultCode> errors should not typically occur if the interface is functioning correctly and will never be accompanied by an indication that the Notification Request should be retried. If you get a large numbers of these errors, you should debug your application.

The Notification Service provides you with the status of your Notification Request as it is processed by the mBlox platform and operator.

Whether your Notification Request fails to validate at the mBlox platform or the operator successfully sends the MT message, your registered callback URL will receive Notification Service XML documents with appropriate status codes that you can evaluate and process.

This XML document consists of a header, providing account identification/authentication, followed by a list containing the delivery status information elements.

  • buffered smsc - Indicates that the message has not yet been delivered due to some operator-related problem but is being retried within the network.
  • acked -Indicates that the message has been accepted by the operator.
  • Delivered - The message was delivered.
  • Non Delivered - See the Delivery status codes referenced below for all possible failure codes
  • Lost Notification - mBlox cannot determine whether this message has been delivered or has failed due to lack of final delivery state information from the operator.

See how these statuses relate to the delivery status codes specified in the <Reason> element and all the possible values in the Delivery status codes reference documentation.

XML structure: Response service¶

When a subscriber sends an SMS message to your account via the operator network, the mBlox platform converts the MO message to XML and sends it to your registered callback URL in a Response Service XML document.

This document consists of a header, providing account identification/authentication, followed by the message being submitted to your application. Although the format of the XML supports multiple MO messages in a list, the mBlox platform will only submit one message per XML document.
